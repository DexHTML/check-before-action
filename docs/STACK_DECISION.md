# Stack decision

## Текущее решение

Для MVP используется статический стек:

- HTML
- CSS
- JavaScript
- GitHub Pages

## Почему так

На первом этапе проект проверяет не технологию, а спрос:

- будут ли страницы индексироваться;
- появятся ли показы;
- будут ли клики;
- будут ли люди запускать инструменты;
- какие темы дадут первые сигналы.

Для этой задачи статический сайт лучше, потому что он:

- бесплатный;
- быстро публикуется;
- не требует сервера;
- не требует базы данных;
- не хранит пользовательские данные;
- меньше ломается;
- проще для индексации;
- безопаснее для срочного MVP.

## Что GitHub Pages реально отдаёт

GitHub Pages публикует статический результат: HTML, CSS, JavaScript и файлы, которые уже лежат в репозитории или были собраны в статический сайт.

Это значит, что можно использовать:

- обычные HTML-страницы;
- CSS;
- JavaScript в браузере;
- статические генераторы через GitHub Actions;
- Jekyll;
- Astro / Vite / React / Vue / Svelte, если они собираются в статические файлы.

Но нельзя использовать как серверное приложение прямо на GitHub Pages:

- PHP в runtime;
- MySQL/MariaDB;
- Python backend;
- Node.js backend;
- личные кабинеты на сервере;
- оплату с серверной проверкой;
- хранение заявок в базе.

## Почему не React/Vue/Astro сразу

React/Vue/Astro можно использовать, если собрать проект в статические HTML/CSS/JS. Но для текущей задачи это пока лишний слой.

Минусы раннего перехода:

- больше файлов и сборки;
- выше риск ошибок;
- сложнее вносить точечные SEO-правки;
- тяжелее сопровождать без локальной разработки;
- нет пользы для проверки спроса на первых 10–20 страницах.

## Когда переходить на генератор

Переход на статический генератор будет оправдан, если:

- страниц станет 30–50+;
- начнёт раздражать ручное дублирование header/footer;
- понадобится единый шаблон страниц;
- захотим хранить контент в Markdown/JSON;
- появится регулярный выпуск новых материалов.

Лучшие кандидаты:

- Eleventy — простой статический генератор для контентного сайта;
- Astro — если нужен красивый современный фронтенд;
- Jekyll — нативно близок к GitHub Pages, но менее удобен для нашей будущей логики;
- Vite — если появится много интерактивных компонентов.

## Когда переходить на PHP

PHP нужен не для SEO-теста, а для продукта:

- база данных;
- админка;
- генерация PDF;
- сохранение проверок;
- оплата;
- личные кабинеты;
- Telegram-интеграции;
- коммерческие комплекты;
- нормальный домен и хостинг.

Переход на PHP/VPS делаем только после первых признаков спроса.

## Текущий вывод

Для текущего этапа HTML/CSS/JS на GitHub Pages — правильный выбор.

Он даёт максимум скорости, минимум риска и не мешает будущему переносу. Если проект покажет спрос, текущие страницы можно перенести в PHP-шаблоны или в статический генератор без потери контента.
